The Static Web
  • Static pages
  • Backend: Apache
The Generated Web
  • Static web pages that are generated/assembled
  • Associated ideas: templating, LAMP stack
  • Backend: Apache+CGI, PHP, Perl, Python
The Read/Write Web
  • Web sites become views of dynamic backend content
  • Associated ideas: Web 2.0, DHTML, AJAX, MVC, blogs/RSS, wikis, Semantic Web, Push, social web, crowd-sourcing (wikipedia).
  • Backend: WSGI, Ruby on Rails, Django
Web Applications
  • Applications delivered via the web but pull/push data from many sources (HTTP, WebSockets, media devices, local storage)
  • Associated ideas: HTML5, mobile, cloud, 'native' app vs webapp.
  • Backend (optional): Node.js, "cloud", big data, NoSQL (CouchDB, MongoDB).

From Here to There

"Native" App vs Web App

What's the Difference?

"Native" App vs Web App

Compiled Object Code vs Interpreted Source?

Nope.

Python/perl/ruby programs are native but interpreted.

Android apps are Dalvik bytecode; somewhere in the middle.

Javascript is a 'compile' target for many languages.

Google's NaCl projects allows compiled code in web apps

"Native" App vs Web App

Installed vs Loaded/Browsed On Demand?

Nope.

Web Apps can work offline.

Some native apps update (parts of) themselves on startup.

"Native" App vs Web App

Load in browser vs Click on Icon?

Still not.

Web apps can run standalone or from shortcuts (with no browser UI)

Native apps can be launch from web links.

"Native" App vs Web App

Integration with Native APIs?

Closer ... but no.

Phonegap/Cordova/AppMobi blur the line.

Everyone except Apple is standardizing on web apps as the new "native" (with access to native APIs):

  • Windows Metro
  • webOS
  • Tizen
  • Blackberry WebWorks
  • Mozilla Boot to Gecko
  • Gnome Shell
  • http://blog.ometer.com/2008/08/25/embeddable-languages/
  • Ubuntu WebApps

Browser extensions provide expanded native API access:

  • Mozilla XUL
  • Chrome Extensions

"Native" App vs Web App

App Marketplace vs Web Download?

Nope.

Chrome Web Store is for web apps.

Mozilla Marketplace will be generic web app marketplace.

PhoneGap/Cordova can package up web apps for Android/Apple app markets

"Native" App vs Web App

Walled Gardens vs Wild Prairies

"Native" means somebody else controls distribution, discovery and monetization.

Example: iOS apps are in Apple's walled garden

  • Pros: high signal to noise, easier to monetize.
  • Cons: Apple takes a cut, beholden to Apple's whims.

Users think of "Native" apps as products. Web apps are more like services (kind-of).

Web apps are harder to sell as a product, but easier to monetize via advertising.

Richness vs Reach

"Native" Apps Have Richness

"Native" apps can maximize the richness of the user experience.

Able to take full advantage of the hardware.

Hardware and software target platforms are finite, knowable and testable.

Least common denominator factor is less of an issue.

Web Apps Have Reach

Number of people/platforms that can be reached is maximized

Smartphones will always have web browsers, therefore "Native" app reach will always be a subset of web app reach.

Limitless variations possible of hardware and browser versions

Do you go after "least common denominator" or sacrifice your reach?

Final Thoughts

HTML5 Application Fitness

Good Match

Content and media delivery (HTML5 has become one of the best design platforms)

Casual games (non-twitch)

Apps with tight social network integration

Not Yet A Good Match

Device specific software/utilities: camera, contacts, etc

Twitch games

Mostly offline apps (soon but not yet)

Sophisticated animation/interaction requirements (however the line is moving quickly)

Consider Progressive Enhancement

  1. Develop Web App First

  2. Use PhoneGap/Cordova or AppMobi to enter "native" marketplaces

  3. When you encounter richness barriers, convert sensitive parts to "native" languages, maintain core design/content rendering in a UIWebView (or equivalent)

  4. If necessary, convert whole app to the "native" language.

Use the Source

The view source menu is responsible for much of the success of the Web. Defacto Open Source

Don't reinvent the wheel. There are many thousands of Javascript libraries and frameworks (see jswiki) and most are under permissive license Open Source libraries. Use them!

Thoughts on Mobile HTML5 Apps

  • The Web is improving extremely rapidly.
    The next 1-2 years will see the gap between web apps and "native" marketplace delivered apps narrow substantially.
  • Adobe has been bitten by the HTML5 bug and is now a major contributor.
    Look for excellent HTML5 content production and workflow tools to arrive in force.
  • On the other hand, watch out for Apple.
    Initially a key driver of the Web, they may begin to consider web apps a threat to their marketplace.
  • Android is currently a problem.
    Even the latest Android version lags in web features.
    However, Chrome for Android is coming soon. Possibly as default in next major release.
  • In app purchases can be much more lucrative than initial app sale.
    Consider the case of TF2.
  • There are several ways of enabling in-app purchases with web apps.
    • PhoneGap/Cordova have some native in app purchase options.
    • AppMobi claims to have seamless native in app purchase.
    • Chrome Web Store has in app purchases.
    • Mozilla Marketplace has in-app purchase capability.
  • Keep an eye on Boot to Gecko and Chrome OS
